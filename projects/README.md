## 一、项目愿景

### 我们为什么做这件事

在监狱工作中，**规范合法的笔录是最好的自我保护**。

当一个事件发生、有据可查时：
- 民警能够证明自己依法履职
- 罪犯无法无理申诉
- 监管部门能够追溯真相

但现实中存在这样的困境：

> 一个民警按规定做了笔录，但格式不规范、内容不完整，最终反而面临质疑。
> 另一个民警做了完整规范的笔录，却因为管理混乱导致记录丢失。
> 还有的情况是——笔录被人篡改，真相被掩盖。

**PrisonSIS 想要解决这些问题。**

### 我们相信

1. **规范即保护** — 一份格式规范、内容完整、签名完整的笔录，是民警最好的护身符
2. **去中心化即可靠** — 没有单一服务器意味着没有单点故障，数据保存在每个人手中
3. **技术应为善** — 加密保护数据不被篡改，签名确保来源可验证，这一切都是为了让真相不被掩盖

### 目标用户

- 监狱一线民警（记录员）
- 监区管理民警
- 监狱信息科技术人员

## 二、系统设计理念

### 2.1 中心化 vs 去中心化

传统笔录系统依赖服务器，一旦服务器故障或网络中断，工作陷入瘫痪。

PrisonSIS 采用**去中心化架构**：

```
传统方案：
  民警终端 ←——网络——→ 服务器（单点故障）
                              ↓
                           数据丢失风险

PrisonSIS：
  民警A（本地SQLite）─┐
  民警B（本地SQLite）─┼── P2P内网 ──→ 扫描件密文备份，无单点
  民警C（本地SQLite）─┘
```

每台电脑既是使用者，也是数据存储节点。**监狱内网本身就是最大的保障**，没有外部攻击风险。

### 2.2 笔录状态机：真相只有一个

```
新建 → 草稿 → 纸质版签字 → 扫描上传 → 最终答案
         ↑                          ↓
      多人可同时编辑                 加密存储
      本地草稿箱                    不可篡改
```

**这是整个系统最核心的设计理念：**

> 草稿是开放的——可以多人协作，可以随意修改
>
> 扫描件是封闭的——一旦生成，就是真相

**为什么这样设计？**

因为在实际工作中：
- 草稿阶段的改动不会造成任何实质影响，只是电子版本
- 最终有法律效力的是纸质版上所有人的亲笔签名和手印
- 数字扫描件只是把这个纸质版安全地保存下来

### 2.3 纸质签名 vs 数字签名

数字签名可以被质疑："系统有没有被黑？"

纸质签名加扫描件：

```
纸质版（有亲笔签名+手印）
    ↓
扫描仪 → PDF
    ↓
AES-256 加密存储（防泄露）
    ↓
SHA-256 哈希（证明未被篡改）
    ↓
RSA-2048 数字签名（证明来源可验证）
```

**双重保障：** 既有纸质版的物理签名，又有数字加密防篡改。

## 三、功能模块

### 核心业务

| 功能 | 说明 |
|------|------|
| 用户认证 | 用户名密码登录，RBAC 角色权限 |
| 罪犯信息管理 | 罪犯档案、编号、案件信息 |
| 笔录制作 | 富文本编辑、模板填充、自动保存草稿 |
| 扫描件上传 | 导入扫描 PDF，自动 AES-256 加密存储，生成哈希和 RSA 签名 |
| P2P 扫描件同步 | 从其他工作站节点下载加密扫描件并解密导出 |
| 审批中心 | 笔录审核（通过/驳回） |
| 档案管理 | 已归档笔录的查询、预览和导出 |
| 审计日志 | 所有关键操作不可删除记录 |

### P2P 协作（去中心化）

| 功能 | 说明 |
|------|------|
| 局域网节点发现 | UDP 广播（端口 25555），自动发现，无需配置 |
| 扫描件密文备份 | 加密扫描件可在多节点存储，互相备份 |
| 扫描件索引广播 | 上传扫描件后广播签名通知，其他节点记录备份索引 |
| 按需拉取扫描件 | 从任意在线节点拉取加密扫描件，用私钥本地解密 |
| 公钥身份体系 | 每人持有 RSA-2048 公私钥对，身份可验证 |

## 四、技术架构

### 4.1 技术栈

| 组件 | 技术选型 |
|------|---------|
| 界面 | Qt6 + C++17 (QWidgets) |
| 数据库 | SQLite + 应用层 AES-256 加密（字段级） |
| 对称加密 | OpenSSL AES-256-CBC |
| 非对称加密 | OpenSSL RSA-2048 |
| 数字签名 | SHA-256 + RSA-2048 |
| 网络 | Qt Network (QUdpSocket + QTcpSocket) |
| PDF | Qt PDF Writer |

### 4.2 加密设计

```
文件加密（AES-256）：
  明文 PDF → AES-256 加密（随机密钥） → 密文存储
    ↑
  AES 密钥用上传者 RSA 公钥加密，嵌入密文文件

哈希验证（SHA-256）：
  原 PDF → SHA-256 → 哈希值（存于数据库）

数字签名（RSA-2048）：
  SHA-256哈希 → RSA-2048私钥签名 → 签名值（防伪造）
```

### 4.3 数据库本地存储

```
~/.local/share/PrisonSystem/PrisonRecordTool/
├── prison_record.db    # SQLite 数据库（笔录内容已 AES-256 加密）
├── keys/{userId}/      # RSA-2048 密钥对（私钥由 PIN 加密）
├── p2p/scans/          # 加密扫描件密文
├── p2p/indices/        # 扫描件索引
└── config/master.key.enc  # 数据库主密钥（由 PIN 保护）
```

**数据完全保存在本地，不上传任何外部服务器。**

## 五、笔录签名完整流程

```
步骤 1：民警在程序内编辑笔录 → 保存草稿
        ↓
步骤 2：点击"打印笔录" → 生成 PDF → 打印
步骤 3：纸质版 → 民警签字 + 罪犯签字 + 罪犯按指模
步骤 4：民警用扫描仪扫描纸质版 → 得到 scan.pdf
步骤 5：程序内点击"🔒 上传扫描件" → 选择 scan.pdf → 输入 PIN
        ↓ 自动完成：
        ① AES-256 加密存储
        ② SHA-256 计算哈希
        ③ RSA-2048 私钥签名
        ④ P2P 网络广播扫描件索引
        ↓
笔录状态变为"已签名扫描件"
        ↓
其他节点记录：此笔录已签名，有法律效力
        ↓
如需在其他工作站获取：点击"📥 下载扫描件" → P2P 拉取密文 → 输入 PIN 解密 → 导出 PDF
```

## 六、安全设计

| 威胁 | 应对措施 |
|------|---------|
| 数据泄露 | AES-256 加密本地数据库字段 |
| 数据篡改 | SHA-256 哈希验证，RSA 签名防伪 |
| 身份伪造 | RSA-2048 公私钥体系，私钥本地 PIN 保护 |
| PIN 遗忘 | setup() 后自动验证，设置失败立即报错 |
| 传输窃听 | P2P 传输内容为加密密文，无私钥无法解密 |
| 否认抵赖 | 数字签名 + 纸质原件双重证据链 |
| 源码安全 | 开源代码（GitHub），接受安全审计 |
